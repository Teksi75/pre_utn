## Addendum — Audit-finding live-removal corrective batch (this run)

### Scope binding (audit findings, this run only)

- **Finding 4 — reachable empty listbox on live-skill removal.** The
  previous `SKILLS_BY_UNIT` constant was captured at module-load
  time: even after the per-render readiness-map fix, a populated
  unit's React `selectedUnit` state can outlive its live skill
  array (HMR or test fixtures mutating the catalog between renders).
  Under that scenario the `<select>` value stayed pinned to the
  unit, but `skillsForUnit.length === 0`, and the render-time
  filter-less `<div role="listbox">` rendered an empty listbox.
  Fix: derive an `effectiveSelectedUnit` on every render from
  `SKILLS_BY_UNIT[unit].length > 0`; use it for the `<select value>`,
  the `skillsForUnit` derivation, and the listbox render gate.
  Replace the frozen module-level `SKILLS_BY_UNIT` with a per-render
  frozen object whose properties are `get` accessors so closures
  captured by `handleUnitChange` and the readiness IIFE read the live
  `UNIT_*_SKILLS` exports each time. No new URL parameter,
  localStorage key, persistence seam, or `setSelectedUnit` reset on
  the down-render path is introduced. If the live catalog
  repopulates on a later render, the original selection is restored
  automatically through `effectiveSelectedUnit`. The
  defensive-selection contract remains the authoritative boundary on
  the IN path; this derivation is the authoritative boundary on the
  OUT path.

- **Finding 5 — STATUS.json unavailable-unit fallback promise.** The
  U5-01 STATUS entry promised "render the exact unavailable-unit
  fallback when the selector's state lands on a zero-skill unit" —
  that contract was retired in `79234e2` in favour of defensive
  selection. The summary text was stale. Fix: rewrite the summary
  to state the delivered contract — per-unit count-derived
  availability, defensive `handleUnitChange` rejection, and a
  per-render `effectiveSelectedUnit` derivation that makes an empty
  live-skill count collapse the listbox rendering. Other STATUS
  fields (`status`, `branch`, `issue`, `startedAt`, `delivery`,
  archive status) are unchanged.

- **Finding 6 — review-size budget.** Per the audit mandate,
  review-size budget is NOT addressed with a fabricated exception
  in this run. The Phase 4 + Phase 5 + Phase 7 audit batches remain
  size-bounded (focused on the FocusSelector presentation layer) and
  fit comfortably under the 400-line budget.

- **Finding 7 — verify-report.md is NOT updated by this batch.**
  The verification report is stale against this commit and will be
  regenerated by an independent verify run after this commit lands.

### Deviations from design

None. The reduced design now explicitly states:

> "Selected-unit live-skill removal invariant | Add a `useState` setter / URL parameter / localStorage key to clear `selectedUnit` on the down-render path, OR derive an `effectiveSelectedUnit` per render from the current `SKILLS_BY_UNIT[unit].length`. | Derive per render. Once a populated unit is selected, `selectedUnit` React state outlives the live catalog contents — HMR or test fixtures can empty that unit's array between renders, and a render-time mutator would still surface an empty listbox on the down-render. A per-render `effectiveSelectedUnit = selectedUnit !== null && SKILLS_BY_UNIT[selectedUnit].length > 0 ? selectedUnit : null` makes the empty condition vanish from the rendering: `<select value>`, `skillsForUnit`, and the listbox gate all observe the live count. The original selection is restored automatically if the live catalog repopulates on a later render. No new URL parameter, localStorage key, persistence seam, or `setSelectedUnit` reset on the down-render path is introduced."

and the data-flow diagram names the per-render frozen getter-based `skillsByUnit`, `effectiveSelectedUnit`, and the recovery arrow from "live repopulates" back to "listbox with current skills". The implementation matches both decisions exactly:

- `skillsByUnit` is a frozen per-render object whose each property is a `get` accessor; each call returns the live `UNIT_X_SKILLS` array.
- `effectiveSelectedUnit` is a per-render derivation; `selectedUnit` React state is never mutated by the live-removal path.
- `<select value>`, `skillsForUnit`, and the listbox render gate all read `effectiveSelectedUnit`.
- No `?unit=` URL parameter, no `localStorage` key, no persistence seam, no `setSelectedUnit` reset on the down-render path, no `analyzeRequestedUnit`, no `UnitRequestAnalysis`, no `UnavailableUnitBanner`, no `UNAVAILABLE_UNIT_MESSAGE`.
